What I was making here was something called the Texture Atlas.
For the sake of efficiency, games like Minecraft stitch all of the textures for each block
together into one big image like this, called the Texture Atlas, so that the renderer doesn't
have to switch between textures every time it draws a different block.
The renderer uses the Atlas image by taking the offset of the texture for the block being
drawn in the Atlas, and then mapping its pixel space coordinates in this 256 by 256 pixel
image into the 0 to 1 UV coordinate space.
Those are then the texture coordinates that are used for the block.
Here we can finally see that working with a single grass block.

This involves a bit of infrastructure setup, since it will be really inefficient to treat

Minecraft uses a system of chunking, where each chunk is 16 by 16 by 256 blocks, and I
decided to do something similar.

You can see the wireframe looks a lot better now, and we're not drawing anything that the

Generally, the way procedurally generated games like Minecraft do terrain generation
is through something called noise, which is a function that takes in a specific number
of parameters corresponding to its dimension, in our case two, and produces a random but
smooth map of values in between negative one and one, which can be mapped into terrain features.
In order to generate more than just a smooth height map, techniques of layering noise like
octave noise and combined noise can be used to produce more interesting terrain features
like I made here.
I decided to follow up for Minecraft Classic on this respect and sample and layer noise
the same way that it did because I thought it fit the vibe pretty well with the bright
green grass and everything.
So I didn't use any real Minecraft lighting in my clone, but there is a quick hack that

the east and west ones in the top and bottom.
This was done by giving each vertex some extra color information in addition to its texture
information, so when OpenGL is calculating the final color for each block vertex, the
vertex is multiplied by the RGB value of the texture.
And as you can see here, that makes a big difference to how the terrain looks because
we can see individual cubes now instead of just big masses of the same block.
The next thing that the terrain really needed was water.
Water is actually pretty simple to add to the terrain generation.
We just pick a water level, I said 64 blocks, and then if the terrain height ends up being

We can see here that that simple addition gives some pretty nice results, but the one

Transparency and translucency are notoriously hard problems in 3D graphics, but the first
thing I had to do to get this right was to start separating out each chunk's mesh into
the solid geometry and the transparent and translucent geometry.
That way, I could have two different sets of vertices that I could render differently.
The key to good looking transparency in 3D graphics is proper ordering, and you can
see here how order matters.
We have four translucent squares, in the left side the red one is on top of the green one,
and on the right side the green one is on top of the red one.
These different orderings give different colors where the squares overlap, so it's important
to make sure that the rendering order corresponds to the physical order of the squares.
This extends out into 3D space, so we need to make sure that the player sees the correct
ordering of block faces, but that requires drawing all of the translucent faces from
back to front relative to the player, meaning they need to be sorted.
This can create issues in Minecraft like games, because there are so many different cube faces
that it gets pretty expensive to sort them all every time a frame needs to be drawn.
I made the mistake of trying to take a lot of shortcuts around this for performance reasons,

To start the next day, I gave up on all of my shortcuts and decided on a proper sorting
algorithm to sort the block faces.
By sorting every transparent or translucent block face inside of each chunk from back
to front relative to where the camera was, this solved all of my issues.
But it did create a few performance problems initially.
So to combat the performance issues, I came up with a few criteria about when to sort
the faces in each chunk.
The faces in the chunk the player is in are sorted whenever the player moves between blocks,
and the faces in the chunk the player is in in every chunk around that chunk are sorted
when the player moves between chunks.
This leads to a good trade off between proper transparency and actual playability.

And even though this atlas is static, there's a trick that we can use to get animations

The key is to duplicate the texture atlas into the number of animation frames that we

blocks texture in the same location on the atlas.
If we number the animation frames for the water block on each texture atlas and play

By swapping out the texture atlas five times a second like this, the world doesn't need

change.
